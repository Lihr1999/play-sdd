## 三、服务端接口与玩法联动测试

### 3.1 接口说明

| 接口 | 方法 | 说明 |
|------|------|------|
| BFF | `GET /api/game/tasks` | 查询参数：`userId`、`campaignId`；经鉴权代理至 Service |
| BFF | `POST /api/game/tasks/:taskId/claim` | Body：`userId`、`campaignId` |
| Service | `GET /game/tasks` | 返回 `TaskWithProgress[]` |
| Service | `POST /game/tasks/:taskId/claim` | 返回 `ClaimResponse` 或 4xx |
| 玩法链 | `POST .../play` | 响应扩展 `taskUpdates: TaskUpdate[]` |

被测核心逻辑：玩法结束后按 `trigger`（`play`|`win`）与是否中奖更新进度；`status=claimed` 不再累计；达到 `target_count` 置 `completed`。

### 3.2 核心判断逻辑（伪代码）

```
after_play(userId, campaignId, prize):
  for task in tasks(campaignId):
    if task.status == 'claimed': skip
    matched = (task.trigger == 'play') OR (task.trigger == 'win' AND prize != null)
    if not matched: skip
    increment current_count; if >= target_count => completed
  return taskUpdates[]
```

### 3.3 测试设计方法

玩法内任务更新：**判定表**（条件组合）；列表查询、领取：**等价类** + **错误推测**（404/409/库存）。

### 3.4 判定表（玩法触发任务进度）

| 规则 | 任务 trigger | 本次玩法是否中奖 (prize) | 任务当前 status | 预期：是否累计进度 |
|------|----------------|---------------------------|-----------------|-------------------|
| R1 | play | 无/有 | in_progress / completed 未满 | 是 |
| R2 | win | 有 | in_progress | 是 |
| R3 | win | 无 | in_progress | **否** |
| R4 | 任意 | 任意 | claimed | **否** |
