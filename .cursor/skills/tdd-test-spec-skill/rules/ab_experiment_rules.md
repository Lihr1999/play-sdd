# A/B 实验功能测试方法论

当需求涉及 **A/B 实验分组控制的 UI/交互变更** 时，使用以下方法论。

## 识别信号

满足以下任一条件，切换为 A/B 实验测试模式：
- 需求存在实验参数（如 `exp_key=0/1`）控制功能开关
- 对照组与实验组的交互行为不同（如：是否展示数字、是否唤起浮层）
- 需求描述"某路径进入不受本次逻辑影响"

## 用例设计四步法

**第一步：实验分组验证**
- 正向：命中实验组 → 展示新特性
- 反向：命中对照组 → 不展示新特性，保持原样式
- 兜底：非法参数值（如 -1/abc）→ 兜底对照组行为
- 稳定性：多次刷新实验分组不变

**第二步：核心功能验证**

逐一枚举所有数据形态（等价类划分）：
- 数值边界（如：0、1~9999、≥10000）各自出用例
- 字段缺失/null 出独立用例验证兜底

**第三步：交互行为对比（对照组 vs 实验组）**

对每个可交互控件，分组出用例：

| 场景 | 对照组用例 | 实验组用例 |
|------|-----------|-----------|
| 首次点击 | icon 状态变化、不展示数字、不唤起浮层 | icon 状态变化、数字+1、不唤起浮层 |
| 再次点击（取消） | icon 恢复原样 | icon 恢复原样、数字-1 |

> 注意：对照组和实验组的不同之处必须分别出独立用例，不可合并。

**第四步：页面覆盖范围（三层分类，必须全部覆盖）**

| 层级 | 类型 | 用例方向 |
|------|------|---------|
| 第一层 | 本次逻辑适用页面（应展示新特性） | 正向验证，多路径可合并为一条用例 |
| 第二层 | 其他场景特殊说明（有特定限制） | 单独出用例，注明限制条件 |
| 第三层 | 不需要本次逻辑（反向验证） | 必须出反向用例，验证不展示新特性 |

## 特殊场景处理规则

**「本来就有不处理」的组件/浮层**
- 必须生成反向验证用例
- 预期结果：不展示新特性，保持原有样式
- 示例：商卡列表浮层 → 用例标题可写「商卡列表浮层—不展示想要人数（本次不处理）」

**跨实验依赖**
- 当 UI 由其他实验控制、交互由本次实验控制时，必须在用例前置条件中注明两个实验参数
- 预期结果分两条：1. UI样式不变（由 X 实验控制）；2. 交互按本次逻辑

**多路径合并**
- 相同预期结果的多个入口路径可合并为一条用例
- 合并方式：测试数据列枚举所有路径，执行步骤用「分别从…进入」

## 覆盖度自查（A/B 实验专项）

- [ ] 实验组/对照组/非法值三种分组状态是否均有用例？
- [ ] 点击交互是否分对照组和实验组分别出用例？
- [ ] 页面覆盖是否按「适用/特殊/不适用」三层分类？
- [ ] 「不处理」的组件是否有反向验证用例？
- [ ] 跨实验依赖是否在前置条件中注明两个实验参数？
- [ ] 数值展示的边界情况（0、中间值、万级）是否逐一覆盖？
