# 测试模式选择

生成测试用例前，先根据需求类型确定测试模式，再读取对应规则文件。

## 决策表

| 识别信号 | 测试模式 | 规则文件 | 输出文档结构 |
|---------|---------|---------|------------|
| 需求存在实验参数（如 `exp_key=0/1`）控制 UI/交互 | **A/B 实验测试** | `rules/ab_experiment_rules.md` | 客户端文档结构 |
| 接口字段值由多个条件组合决定 / 测试方案说明通过 mock 构造输入 | **服务端接口测试** | `rules/server_interface_rules.md` | 服务端接口文档结构 |
| 客户端 UI 展示、页面跳转、表单交互、后台管理等 | **客户端功能测试**（默认） | `rules/generation_rules.md` | 客户端文档结构 |

> **优先级**：A/B 实验 > 服务端接口 > 客户端功能（默认）

## 混合场景处理

| 场景 | 处理方式 |
|------|---------|
| A/B 实验中含接口判断逻辑 | 以 A/B 实验模式为主，在实验分组验证模块内补充接口边界用例 |
| 服务端接口中含实验参数控制 | 以服务端接口模式为主，接口字段用判定表覆盖，实验参数作为其中一个条件列 |
| 客户端功能涉及后台管理页面 | 客户端功能测试模式，后台页面补充 `generation_rules.md` 中的「通用用例」 |

## 输入为 Superpowers Plan 时

- 先在文档内检索：**exp_key / 实验组 / variant** → 倾向 A/B；**判定表 / Mock / 多条件组合决定字段** → 倾向服务端接口；其余默认客户端功能。
- 单体 Plan 常同时含 web / bff / service：在「测试概述 → 测试范围」中按层列出；各层用例仍按上表对段落分别判定（同一 Markdown 中可出现多个「## 二、{模块}」结构，模块名可带层级如「Service 游戏接口」）。

## 输入为 OpenSpec `spec.md` 时

- 大量 **SHALL**、HTTP 路径、`POST`/`GET`、状态码、错误体 → 倾向 **服务端接口测试** 文档结构；若同一文件仅含 UI/皮肤/主题要求则仍走客户端。  
- 文中已给 **Scenario / WHEN / THEN** 时，用例以之为骨架，再按 `generation_rules.md` 补全反向与操作后状态。
